# Trader Risk Radar — 構想討論紀錄

> 本文件記錄專案發想階段的完整討論脈絡，作為後續 PRD / 架構設計的背景依據。
> 日期：2026-06-19

---

## 1. 專案緣起

從「想做新創卻沒有痛點」出發，經過幾輪挖掘，最終鎖定一個**作者真實經歷過、且天生屬於 Web3 領域**的痛點：

作者私下在做**鏈上永續合約交易（開倉）**，並曾自製一套**跟單系統**：分析各交易員的交易狀況、排名，再進行跟單。

### 核心痛點

> 跟單的實際結果與預期相反。當被跟的交易員**本金夠大、承受率夠高**時，相對於本金較小的散戶，散戶會**更容易在價格逆向時被洗出場、甚至爆倉**——結果不是跟著交易員賺錢，而是一路賠到出場。

---

## 2. 痛點的根因診斷

### 馬丁格爾陷阱（最可能的根因）

舊系統的排名演算法，很可能**自動篩選出「攤平 / 加倉型（馬丁格爾）」的交易員**，而這種交易員是**最致命、最不該跟**的一種：

1. 開倉 → 價格逆向 → **不停損，反而加倉攤平**（補保證金、拉低均價）。
2. 因本金深，能一路加到價格回來，最後小賺出場。
3. 結果：**勝率超高、回撤看起來都能撐回來** → 在排名系統裡長得像神。

散戶跟單時：本金小、加不了那麼多倉 → 在價格走到最深處（交易員正準備加最後一筆、即將反轉的位置）**先爆倉** → 然後價格反轉，交易員賺錢，散戶已經死了。

> **結論：散戶不是跟著交易員賠，而是死在交易員賺錢的必經之路上。**

### 關鍵洞察：複製錯了東西

跟單最大的迷思是「以為複製進出場點 = 複製策略」。真正該複製卻看不到的是**風險結構**：

| 看得到（被複製了） | 看不到（生死關鍵，沒被複製）          |
| ------------------ | ------------------------------------- |
| 進場價、方向、槓桿 | 倉位佔交易員**總資產**幾 %            |
| 出場價             | 他準備了多少**保證金後備**來加倉      |
|                    | 這套打法的**最大逆向幅度（MAE）**多深 |
|                    | 他有沒有停損，還是死扛                |

本金差距 = 水面下撐住倉位的那座山的大小差距。

## 3. 產品形狀

**不做代操、不自動下單**，而是做一個：

> **針對鏈上交易員的「分析 + 風控工具」，提供風險排行與即時訊號。**

使用者透過 API / CLI 指令向專案查詢，專案回傳訊號與分析結果。

### 為什麼這個形狀對

- 避開最麻煩的部分：代操、資金託管、法規、私鑰管理。
- 切入市面跟單平台的縫隙：平台賺手續費，**不在乎使用者會不會被洗出場**；風控工具正好補這個缺。
- 產品化形態應為**賣分析 / 風控工具給散戶（SaaS / 訂閱）**，讓使用者自己決策；**不是**幫人代操（法規雷區）。

### 核心差異化功能（解決上述根因）

1. **用「風格」分類交易員，而非只用報酬排名** — 標記停損型 vs 攤平死扛型、穩定倉位 vs 越輸下越大；將攤平型過濾或標為「高危」。
2. **量化每個交易員的最大逆向幅度（MAE）** — 平均要扛多深浮虧才回本/獲利，直接決定跟單所需的緩衝與後備金。
3. **倉位大小用「我能承受的爆倉距離」回推，而非複製對方槓桿** — 用風險對齊，不用名目金額對齊。
4. **偵測加倉 / 攤平行為** — 預先分配後備金一起加，或直接判定「複製不起，跳過」。

---

## 4. 誠實的風險認知

- **「能解釋」不等於「能賺錢」**：鏈上合約是負和遊戲（手續費、資金費率、滑價、MEV）。完美修正後，可能做出的是「**輸得比較慢、比較不會爆倉**」的系統，離穩定獲利仍有距離。**不要把身家壓進去驗證。**
- **產品化最大難點是信任，不是技術**：所以定位在「賣分析/風控工具」而非代操。
- **期望管理**：以零碎時間投入，最務實的目標是「做出自己天天用的爽工具」，財富自由當 bonus。**持續**才是零碎時間創業唯一的勝算。

## 5. 技術決策（已收斂）

### 先決決定：資料來源聚焦單一協議

- **第一版只做 Hyperliquid**（目前最大鏈上永續合約）：官方免費 WebSocket + REST API，可直接拿到每地址逐筆成交、持倉、資金費率。
- 不做「全鏈通吃」（創業常見死法）。驗證有效後再擴張到 GMX / dYdX / Gains 等。

### 資料攝取策略：定時輪詢（polling），非串流

- **分析深度**：polling 完全足夠。倉位通常開數小時到數天，每 15~30s 拍一張快照即可還原浮虧曲線。
- **進場訊號時效**：polling 有最多一個輪詢間隔的延遲，但本產品是分析工具、給訊號讓使用者判斷，非搶秒級進場，可接受。
- **唯一掉資料處**：交易員在兩次輪詢間開倉又平倉（intra-interval round trip）會看不到——但這種快進快出單本來就不是目標客戶想跟的，無妨。
- **好處**：省掉常駐 WebSocket listener，免處理斷線重連/心跳，維運大幅簡化。

### 推薦技術棧（Node.js）

| 層級                    | 選擇                                                   | 理由                                         |
| ----------------------- | ------------------------------------------------------ | -------------------------------------------- |
| 語言                    | **TypeScript**                                         | 碰錢碰大數字，型別安全是命                   |
| 數字處理                | **`bigint` + `decimal.js` / `dnum`**                   | ⚠️ 絕不可用 JS float，精度會爆炸             |
| 鏈上互動（之後 EVM 用） | **`viem`**                                             | TS-first、現代；Hyperliquid 第一版可能用不到 |
| 後端框架                | **Fastify**                                            | 快、TS 支援好；NestJS 對 solo 太重           |
| 排程 / 背景任務         | **BullMQ**（吃 Redis）                                 | 分層輪詢、定期重算排名/MAE                   |
| 儲存                    | **PostgreSQL + TimescaleDB**                           | 資料本質是時序；Redis 當快取/PubSub          |
| 訊號推送                | REST（pull）+ **SSE**（push）                          | SSE 比 WebSocket 簡單，單向推送夠用          |
| 部署                    | 單一 VPS（Hetzner / Railway / Fly.io）+ Docker Compose | 別碰 K8s/微服務                              |

### 架構圖

```
   ┌────────────────────────────────────┐
   │  Scheduler (BullMQ repeatable jobs)  │   ← 分層排程
   │  - 高排名交易員：每 15~30s 撈一次      │
   │  - 長尾交易員：每幾分鐘撈一次          │
   └──────────────┬───────────────────────┘
                  │ 撈 Hyperliquid API
          ┌───────▼────────┐
          │ Ingest Worker    │ 去重後寫入
          └───────┬────────┘
            ┌──────▼──────┐
            │ PostgreSQL   │  ← 原始資料 + 快照
            │ +TimescaleDB │
            └──────┬──────┘
          ┌────────▼─────────┐
          │ 分析引擎 (排程跑)  │ → 算 MAE/風格/排名，寫回 DB
          └────────┬─────────┘
            ┌───────▼────────┐
            │ Fastify API     │  GET /rankings, /signals
            └────────────────┘
```

### 資料模型（分析深度的命脈）

```
traders            (address, first_seen, ...)
positions          (id, trader, coin, side, status[open/closed], ...)
position_events    (position_id, type[open/add/reduce/close],
                    price, size, leverage, ts)        ← 逐筆動作
position_snapshots (position_id, ts, mark_price,
                    unrealized_pnl_pct, margin)        ← 每次輪詢拍一張
trader_metrics     (trader, win_rate, avg_mae,
                    max_drawdown_survived, style_label, ...)  ← 分析結果
```

- **`position_events`**：偵測攤平加倉的鐵證來源（同倉位虧損中出現 size 越加越大的 `add`）。
- **`position_snapshots`**：`MAE = min(unrealized_pnl_pct)`——polling 能算出 MAE 的原理。

> 核心心法：**「定時撈」撈回來的不只是現在持倉，而是把每次狀態當時間切片存下來；分析深度是從這堆切片「長」出來的。**

### 必做工程細節

1. **去重（idempotency）**：用成交唯一 id/hash 當 unique key，否則統計重複計算全錯。
2. **分層輪詢 + 尊重 rate limit**：高排名撈勤、長尾撈鬆；用 Hyperliquid `clearinghouseState` 一次拿一地址完整持倉。

---

## 6. 第一個里程碑（最薄的一條線）

對**單一交易員**跑通：

1. 定時撈該地址資料
2. 存成 `position_events` + `position_snapshots`
3. 算出他的 **MAE** 與**是否攤平**

打通這條，分析引擎核心就活了，其餘皆為擴充。架構與演算法解耦——指標細節（看總報酬 / 勝率 / 盈虧比…）可後續隨時替換。
